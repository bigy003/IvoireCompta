1. Contexte et objectifs
1.1 Présentation du projet
IvoireCompta est un logiciel SaaS de gestion comptable et fiscale destiné exclusivement aux
cabinets d'expertise comptable en Côte d'Ivoire. Il se différencie des solutions génériques (EBP,
Sage) par une conformité native aux réglementations ivoiriennes : SYSCOHADA révisé 2018, 9
tableaux additionnels DGI-CI, intégration e-impôts.gouv.ci, et gestion de la paie selon les règles
CNPS/ITS locales.

1.2 Problèmes résolus
    •   Aucun logiciel du marché ne gère les 9 tableaux additionnels obligatoires DGI-CI (T01 à
        T09) depuis 2023
    •   Le visa électronique ONECCA-CI sur e-impôts.gouv.ci est un processus manuel de 3 à 5h
        par dossier
    •   La réforme DGI 2026 rend obligatoire la digitalisation des états financiers — aucun outil
        local n'est prêt
    •   Les coupures de réseau à Abidjan imposent un fonctionnement offline-first que les
        solutions cloud standard ne supportent pas
    •   Le calendrier fiscal est géré sur Excel dans la plupart des cabinets, source d'erreurs et de
        pénalités



1.3 Périmètre du MVP (Phase 1 — Mois 1 à 3)
Le MVP cible les fonctionnalités strictement nécessaires pour que 2 à 3 cabinets pilotes puissent
utiliser IvoireCompta pour leur clôture DSF d'avril 2026.

2.2 Multi-tenant et isolation des données
Chaque cabinet est un tenant isolé. L'isolation est assurée à deux niveaux :
    •   Application : toutes les requêtes Prisma filtrent sur cabinetId — aucune donnée d'un
        cabinet n'est accessible depuis un autre
    •   Base de données : Row-Level Security (RLS) PostgreSQL activé sur toutes les tables
        sensibles, avec une policy basée sur le JWT
    •   Audit log immuable : toutes les actions critiques (écritures, visa DSF) sont journalisées
        dans la table audit_logs sans possibilité de suppression

2.3 Offline-first (PWA)
Le frontend fonctionne sans connexion grâce à :
    •   IndexedDB pour la persistance locale des écritures en cours de saisie
    •   Service Worker pour le cache des assets et des données récentes
    •   Synchronisation automatique dès rétablissement de la connexion avec résolution de
        conflits (last-write-wins sur les brouillons, validation bloquée si conflit sur une écriture
        validée)

3.2 Précisions sur les types de données
Toutes les valeurs monétaires sont stockées en bigint (entier FCFA). Il n'y a pas de centimes en
FCFA. L'utilisation de bigint évite les erreurs d'arrondi des flottants JavaScript pour les calculs
fiscaux.
Le champ donnees des tableaux DSF est de type JSON (PostgreSQL jsonb). Chaque tableau
(T01 à T09) ayant une structure radicalement différente, le JSON flexible est préférable à 9 tables
rigides. Le codeTableau identifie la structure attendue.

4.2 Calcul IS et IMF
Règle fondamentale CI : l'impôt dû est toujours le maximum entre l'IS calculé et l'IMF, même en
cas de déficit.

6. Sécurité et conformité
6.1 Authentification
    •   JWT avec expiration 8h + refresh token 30 jours
    •   2FA TOTP obligatoire pour les actions visa DSF (OTP lib conforme RFC 6238)
    •   Token de visa à durée limitée 15 minutes — usage unique par DSF
    •   Vérification du numéro d'ordre ONECCA-CI en temps réel avant tout visa

6.2 Chiffrement et protection des données
    •   TLS 1.3 sur toutes les communications
    •   Chiffrement AES-256 des données sensibles en base (totpSecret, informations bancaires)
    •   Documents stockés sur S3 avec URL pré-signées à durée limitée (1h)
    •   Hash SHA-256 des tableaux DSF visés pour garantie d'intégrité

6.3 Audit et traçabilité
    •   Toutes les actions critiques sont journalisées dans audit_logs (immuable — pas de
        DELETE ni UPDATE)
    •   Données avant/après pour toutes les modifications de valeurs comptables
    •   IP address enregistrée pour les actions de visa
    •   Retention des logs : 10 ans (obligation légale comptable CI)
